# エアホッケー（2台のiPad対戦）実装仕様書

## 0. 概要

同じWi-Fiにつないだ2台のiPadで、7歳と5歳のきょうだいが対戦するエアホッケーゲーム。
単一HTMLファイルで完結し、GitHub Pages（https）にデプロイして使う。
通信は PeerJS（クラウドのシグナリングサーバー経由）でP2P接続する。

**この仕様書の対象**: アーキテクチャ・通信・ゲームロジック・操作・状態管理。
**対象外**: ビジュアルデザイン（別途 Claude Design が担当。後からCSS/マークアップを差し替える前提で、ロジックとDOM構造を疎結合に保つこと）。

---

## 1. 技術スタック・制約

- **単一HTMLファイル**（HTML + CSS + JS を1ファイルに収める）
- 外部依存は PeerJS のみ（CDNから読み込み）
  - `<script src="https://unpkg.com/peerjs@1.5.4/dist/peerjs.min.js"></script>`
- フレームワーク不使用（バニラJS）。ビルド工程なし
- 描画は Canvas（`<canvas>`）を使う。台・パック・マレットはCanvasに描画
- GitHub Pages（https）で配信する前提。`file://` や `http://` のローカルIP直アクセスでは WebRTC が動かないため考慮不要
- 対象ブラウザ: iPad の Safari（最新2バージョン程度）
- **横持ち（landscape）前提**

---

## 2. 画面の向き（横持ち）

- 横持ち（landscape）でプレイする
- 縦持ち（portrait）で開かれたら、「iPadを横にしてね」という回転をうながすオーバーレイを表示し、ゲームを進めさせない
  - `window.matchMedia("(orientation: portrait)")` で判定
- Canvasは画面サイズにフィットさせる（リサイズ時・回転時に再計算）
- 横持ちなので、ホッケー台は**左右2分割**。左プレイヤー / 右プレイヤー
  - ゴールは左端と右端
  - 中央に縦の中央線

---

## 3. アーキテクチャ（ホスト権威モデル）

物理演算は**ホスト1台のみが行う**。両者が計算すると座標がズレて破綻するため。

```
ホスト（iPad A）
  - すべてのゲーム状態を保持（パック座標・速度、両マレット座標、スコア、ゲーム状態）
  - 物理演算ループを回す（パック移動・壁反射・マレット衝突・ゴール判定）
  - 毎フレーム、状態をゲストへ送信
  - 自分のマレットはローカルのタッチで操作

ゲスト（iPad B）
  - 物理演算しない
  - 自分のマレット座標をホストへ送信
  - ホストから来た状態（パック・相手マレット・スコア）を描画するだけ
```

### 状態の持ち主
| データ | 持ち主 | 流れ |
|---|---|---|
| パック座標・速度 | ホスト | ホスト→ゲスト |
| スコア | ホスト | ホスト→ゲスト |
| ゲーム状態（待機/プレイ中/得点演出/結果） | ホスト | ホスト→ゲスト |
| ホスト自身のマレット | ホスト | ホスト→ゲスト |
| ゲストのマレット | ゲスト | ゲスト→ホスト |

---

## 4. 通信仕様（PeerJS）

### 接続確立
- ホスト: `new Peer()` で**自動生成IDを使う**（衝突回避のため固定IDは避ける）。発行されたIDを画面に表示し、ゲストに伝える
  - IDは長いので、表示用に短い「合言葉」方式にしたい場合は、4桁の数字コードを生成してホストIDに使う案も可（`new Peer("room-" + 4桁)`）。ただし衝突時の `unavailable-id` ハンドリングを入れること
- ゲスト: ホストのID（または合言葉）を入力し `peer.connect(hostId)` でつなぐ
- `peer.on('open')`（自分の登録完了）と `conn.on('open')`（相手と接続確立）は別物。データ送受信は後者以降

### DataChannel 設定（低遅延優先）
```js
conn = peer.connect(hostId, {
  reliable: false,        // 再送なし。古いパケットで詰まらせない
  serialization: "json"
});
```

### 送信頻度
- マレット座標（ゲスト→ホスト）: `requestAnimationFrame` ごと（最大60Hz）
- 状態同期（ホスト→ゲスト）: 30〜60Hz。負荷が高ければ間引く
- 毎フレーム最新値で上書きする設計なので、パケットが多少落ちても問題ない

### メッセージ形式（例）
```js
// ゲスト → ホスト
{ t: "mallet", x: 0.42, y: 0.61 }   // 座標は 0〜1 の正規化値（解像度差を吸収）

// ホスト → ゲスト
{
  t: "state",
  puck:   { x: 0.5, y: 0.5 },
  mallet: { x: 0.12, y: 0.5 },       // ホスト側マレット
  score:  { host: 2, guest: 1 },
  phase:  "playing"                   // waiting / playing / goal / result
}
```

- **座標は 0〜1 に正規化**して送る（2台の画面解像度・Canvasサイズが違っても破綻しないように）。受信側で自分のCanvasサイズに掛け戻す

### 補間（ゲスト側）
- 受信したパック座標へ瞬間移動させるとカクつく。直前値→受信値を線形補間（lerp）して1〜2フレームで滑らかに追従させる

### 切断・再接続
- `conn.on('close')` / `peer.on('disconnected')` を検知し、「相手が切れました」表示＋再接続導線を出す
- `peer.on('error')` の主要ケースを処理:
  - `peer-unavailable`（ホストIDが存在しない/未起動）
  - `unavailable-id`（合言葉方式で衝突）
  - `network`（PeerJSサーバー到達不可）

## 5. ゲームロジック（ホスト側で実行）

### 物理
- パックは等速直線運動＋壁反射。摩擦はごく弱く（または無し）
- 上下の壁（横持ちなので上下＝長辺）で反射
- 左右の端はゴール以外の部分が壁。ゴール部分に入ったら得点
- マレットとパックの衝突: 円同士の衝突。マレットの移動速度をパックに加える（押し出す感じ）。速度に上限を設けて飛びすぎ防止
- パックの最低速度を保証（止まると試合が進まないため、遅すぎたら少し加速）

### マレットの制約
- 各マレットは自分の陣地（左半分 or 右半分）から出られない（中央線を越えない）
- マレットは台の外に出られない

### 得点・勝敗
- ゴールにパックが入ったら相手側の得点 +1
- 得点後: パックを中央にリセット、短い「得点演出」フェーズ（1〜2秒）を挟んでから再開
- **先取点で勝利**（デフォルト5点。定数で変更可能に）
- 勝利したら結果フェーズへ

### ゲームフェーズ（状態機械）
```
waiting   : 2台が接続するまでの待機
playing   : 対戦中（物理ループ稼働）
goal      : 得点演出中（1〜2秒、パックリセット）
result    : 勝敗確定。もう1回で playing に戻す
```

## 6. 年齢差バランス調整（重要）

7歳と5歳が遊ぶため、ハンデ機能を入れる。**接続画面で「年下はどっち側か」を選べるように**し、以下を年下側だけに適用する（オン/オフ可能に）。

- **ゴール幅**: 年下側のゴールを狭く、年上側のゴールを広くする（年下が守りやすく、攻めやすい）
  - 実装メモ: ゴール幅は左右で別々の定数で持つ
- **マレットサイズ**: 年下側のマレットを少し大きく（パックに当てやすい）

ハンデは「年下が露骨に有利」になりすぎない程度に。デフォルトのハンデ量も定数で調整できるようにしておく。

---

## 7. 操作（タッチ）

- 自分のマレットを指でドラッグして動かす（タッチ位置に追従、または近接ドラッグ）
- 自陣の範囲内に制限（中央線を越えさせない）
- マルチタッチは考慮不要（1本指想定）。複数タッチが来たら最初の1点だけ使う
- `touch-action: none` 等でブラウザのスクロール・ズームを抑止
- iPadのSafariで、ダブルタップズーム・長押しメニュー・バウンススクロールが出ないようにする

---

## 8. DOM構造の方針（デザイン差し替え前提）

Claude Design がビジュアルを作るので、**ロジックとビジュアルを疎結合**にする。

- 画面（接続/対戦/結果/回転うながし）は別々のコンテナに分け、表示/非表示を切り替える
- ゲーム描画は `<canvas>` に集約。台・パック・マレット・中央線・ゴールはCanvas内に描く
- スコア・ボタン・ID表示など**UI要素はCanvasの外のDOM**に置く（後でデザインを当てやすい）
- 色やサイズは CSS変数 / JS定数にまとめ、後から差し替えやすくする
- クラス名・id は意味のある命名にし、デザイン側が触りやすいようコメントを残す

---

## 9. チューニング用の定数（まとめて先頭に定義）

最低限これらは1か所にまとめて、あとから調整できるように:

- 勝利点数（デフォルト5）
- パックの初速・最高速・最低速・摩擦係数
- マレット半径（年上/年下）、パック半径
- ゴール幅（年上/年下）
- 反発係数（マレット衝突時にパックへ伝える速度の倍率）
- 状態同期の送信Hz、補間の強さ（lerp係数）
- 得点演出フェーズの長さ（ミリ秒）

---

## 10. 受け入れ基準（動作確認したいこと）

1. 2台のiPadを横持ちにし、片方がホスト・片方がゲストで接続できる
2. 接続後、両画面で同じ位置にパックが見える（ズレが小さい）
3. それぞれが自分のマレットを動かせ、相手の画面にも反映される
4. パックを打ち合え、ゴールでスコアが入る
5. 先取点で結果画面になり、「もう1回」で再戦できる
6. 縦持ちにすると「横にしてね」の表示が出る
7. 片方を切断すると、もう片方に「相手が切れました」が出る
8. 年下ハンデ（ゴール幅・マレットサイズ）がオン/オフで効く

---

## 11. 補足・注意

- パックやマレットの座標は**正規化（0〜1）してから送受信**すること（2台の解像度差対策）。これを怠ると片方でズレる
- ホストとゲストで物理を二重計算しないこと（ホストのみ）
- 初回接続の手順（誰がホストか、IDをどう伝えるか）が5歳には難しいので、**接続は基本おとなが手伝う前提**でよい。ただしボタンは大きく、間違えにくく
- まずは**1台のブラウザ内でホスト/ゲストの両ロジックが動くか**を確認してから、実機2台に進めると開発が楽（ローカルで2タブ開いてテスト可能）
